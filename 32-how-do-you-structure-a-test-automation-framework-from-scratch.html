QUESTION 32 · TEST AUTOMATION

How do you structure a test automation framework from scratch?

Bạn xây dựng cấu trúc một test automation framework từ đầu như thế nào?

Sample Answer (English)

When I build a test automation framework from scratch, I start with a clear folder structure: a pages directory for page objects, a tests directory for the actual test specs, a fixtures directory for test data and setup logic, and a utils directory for shared helpers like API clients or custom assertions. Next I set up config management, using environment-specific config files or .env variables so the same suite can run against dev, staging, or production. I also plan reporting integration early, wiring up HTML or Allure reports so failures are easy to debug. Then I hook the suite into CI, so every pull request triggers the pipeline automatically. Finally, I choose the right test runner, like Playwright or Cypress, based on the application under test and the team's skill set, and I keep the architecture modular so it can scale as the test suite grows.

🇻🇳 Bản dịch tiếng Việt

Khi xây dựng một test automation framework từ đầu, tôi bắt đầu với một cấu trúc thư mục (folder structure) rõ ràng: thư mục pages chứa các page object, thư mục tests chứa các test spec thực tế, thư mục fixtures chứa dữ liệu test và logic thiết lập, và thư mục utils chứa các hàm dùng chung như API client hay các hàm assertion tùy chỉnh. Tiếp theo tôi thiết lập việc quản lý cấu hình (config management), dùng các file config riêng cho từng môi trường hoặc biến .env để cùng một bộ test có thể chạy trên dev, staging, hoặc production. Tôi cũng lên kế hoạch tích hợp báo cáo (reporting integration) sớm, kết nối báo cáo HTML hoặc Allure để dễ dàng debug khi test thất bại. Sau đó tôi tích hợp bộ test vào CI, để mỗi pull request tự động kích hoạt pipeline. Cuối cùng, tôi chọn đúng công cụ chạy test (test runner), như Playwright hay Cypress, dựa trên ứng dụng đang được kiểm thử và kỹ năng của đội ngũ, đồng thời giữ kiến trúc theo dạng module để có thể mở rộng (scale) khi bộ test lớn dần.

💻 Cấu trúc thư mục mẫu

// Cấu trúc thư mục cho một test automation framework (Playwright)
project-root/
├── pages/                 // Page Object classes
│   ├── LoginPage.js
│   └── DashboardPage.js
├── tests/                 // Test specs (theo feature/module)
│   ├── auth/
│   │   └── login.spec.js
│   └── checkout/
│       └── checkout.spec.js
├── fixtures/              // Test data + custom fixtures/setup
│   ├── users.json
│   └── test-fixtures.js
├── utils/                 // Helper functions dùng chung
│   ├── apiClient.js
│   └── assertions.js
├── config/                // Config theo môi trường
│   ├── dev.config.js
│   └── staging.config.js
├── playwright.config.js   // Config chính: runner, reporter, timeout
├── .env                   // Biến môi trường (base URL, secrets)
└── .github/workflows/
    └── ci.yml             // Pipeline: chạy test trên mỗi PR

✅ Vai trò từng thư mục / file

Thư mục / FileMục đích
pages/Chứa các page object, gom locator và action theo từng trang
tests/Chứa test spec thực tế, tổ chức theo feature hoặc module
fixtures/Dữ liệu test và logic setup/teardown dùng chung
utils/Helper functions: API client, custom assertions, logger...
config/Cấu hình riêng cho từng môi trường (dev, staging, prod)
playwright.config.jsCấu hình runner: reporter, timeout, base URL, projects
.github/workflows/Định nghĩa CI pipeline, tự động chạy test khi có PR
