QUESTION 18 · TEST AUTOMATION

What is visual regression testing, and when do you use it?

Visual regression testing là gì, và khi nào bạn dùng nó?

Sample Answer (English)

Visual regression testing is a technique where I capture a screenshot of a page or component, save it as a baseline image, and then compare future screenshots against that baseline using a pixel diff algorithm. If the difference goes above a configured threshold, the test fails and highlights exactly which pixels changed. In Playwright, I use the toHaveScreenshot() assertion, which handles capturing, comparing, and even updating baselines when the UI intentionally changes. I mainly rely on this for catching unintended CSS or layout changes, things like broken spacing, overlapping elements, or a component silently losing its styling, that functional assertions like checking text or attributes would never catch. I usually run it on key pages, shared components, and after any CSS refactor or design system update.

🇻🇳 Bản dịch tiếng Việt

Visual regression testing là kỹ thuật mà tôi chụp ảnh màn hình (screenshot) của một trang hoặc một component, lưu nó làm ảnh baseline, sau đó so sánh các ảnh chụp sau này với baseline đó bằng thuật toán so sánh pixel (pixel diff). Nếu độ khác biệt vượt quá một ngưỡng (threshold) đã cấu hình, test sẽ thất bại và làm nổi bật chính xác những pixel nào đã thay đổi. Trong Playwright, tôi dùng assertion toHaveScreenshot(), nó xử lý việc chụp ảnh, so sánh, và thậm chí cập nhật baseline khi giao diện thay đổi có chủ đích. Tôi chủ yếu dùng kỹ thuật này để phát hiện những thay đổi CSS ngoài ý muốn hoặc thay đổi layout, ví dụ như khoảng cách bị vỡ, các phần tử chồng lên nhau, hoặc một component bị mất style một cách âm thầm, những điều mà các assertion chức năng như kiểm tra text hay thuộc tính sẽ không bao giờ phát hiện được. Tôi thường chạy nó trên các trang quan trọng, component dùng chung, và sau mỗi lần refactor CSS hoặc cập nhật design system.

💻 Ví dụ minh họa (Playwright)

// visual.spec.js — so sánh ảnh chụp với baseline
import { test, expect } from '@playwright/test';

test('homepage matches baseline', async ({ page }) => {
  await page.goto('https://example.com');

  // Chụp toàn trang, so với ảnh baseline đã lưu
  await expect(page).toHaveScreenshot('homepage.png', {
    maxDiffPixelRatio: 0.02, // ngưỡng chấp nhận khác biệt 2%
  });
});

test('login card matches baseline', async ({ page }) => {
  await page.goto('https://example.com/login');
  const card = page.getByTestId('login-card');
  await expect(card).toHaveScreenshot('login-card.png');
});

// Cập nhật lại baseline khi UI đổi có chủ đích:
// npx playwright test --update-snapshots

⚖️ Khi nào nên dùng / không nên dùng

Trường hợpGhi chú
✅ Nên dùng — trang/component quan trọngTrang chủ, checkout, dashboard — nơi lỗi layout ảnh hưởng UX nghiêm trọng
✅ Nên dùng — sau refactor CSS/design systemPhát hiện side-effect ngoài ý muốn khi đổi style dùng chung
✅ Nên dùng — component thư viện UIĐảm bảo button, form, card... không bị vỡ giao diện qua thời gian
❌ Không nên dùng — nội dung động caoVùng có dữ liệu random, timestamp, ảnh quảng cáo gây false positive liên tục
❌ Không nên dùng — thay cho functional testKhông kiểm tra được logic nghiệp vụ, chỉ kiểm tra hình ảnh
⚠️ Cân nhắcChi phí bảo trì baseline khi UI thay đổi thường xuyên, cần review ảnh diff thủ công
