QUESTION 22 · TEST AUTOMATION

What is shift-left testing, and how have you applied it?

Shift-left testing là gì, và bạn đã áp dụng nó như thế nào?

Sample Answer (English)

Shift-left testing means moving testing activities earlier in the SDLC instead of leaving them until the end, right before release. In practice, I get QA involved during the requirements and design review, so we can catch ambiguous or untestable requirements before any code is written. I also write automated tests alongside development, often using a test-driven or behavior-driven approach, so unit and integration tests run on every commit through the CI pipeline. This tight collaboration with developers means bugs are caught earlier and fixed while the code is still fresh in everyone's mind, which is far cheaper than finding them in a full regression cycle later. Overall, shifting left reduces the cost of fixing defects, shortens the feedback loop, and builds quality into the product from the start rather than inspecting it in afterward.

🇻🇳 Bản dịch tiếng Việt

Shift-left testing nghĩa là đưa các hoạt động kiểm thử lên sớm hơn trong vòng đời phát triển phần mềm (SDLC), thay vì để đến cuối, ngay trước khi release. Trong thực tế, tôi để QA tham gia ngay từ giai đoạn review yêu cầu và thiết kế, để có thể phát hiện những yêu cầu mơ hồ hoặc không thể kiểm thử trước khi bất kỳ dòng code nào được viết. Tôi cũng viết test tự động song song với quá trình phát triển, thường theo hướng test-driven hoặc behavior-driven, để các unit test và integration test chạy trên mỗi commit thông qua pipeline CI. Sự hợp tác với các lập trình viên chặt chẽ này giúp lỗi được phát hiện sớm hơn và sửa khi code vẫn còn mới trong đầu mọi người, điều này rẻ hơn rất nhiều so với việc tìm ra lỗi trong một chu kỳ regression đầy đủ sau này. Nhìn chung, việc dịch chuyển sang trái giúp giảm chi phí sửa lỗi, rút ngắn vòng phản hồi, và xây dựng chất lượng vào sản phẩm ngay từ đầu thay vì kiểm tra nó sau đó.

💻 Minh họa Shift-Left

// TRADITIONAL (test-last) — kiểm thử dồn về cuối SDLC
Requirements → Design → Development → [ TESTING ] → Release
                                          ^
                          // bug tìm ra ở đây rất TỐN KÉM để sửa

// SHIFT-LEFT — kiểm thử trải đều xuyên suốt SDLC
[ Review ]Requirements → [ Review ]Design
      ↓                        ↓
  QA góp ý sớm         QA góp ý sớm

Development  // viết test song song với code
  ├─ Unit tests        [ TESTING ]
  ├─ Integration tests [ TESTING ]  → chạy trên mỗi commit (CI)
  └─ Static analysis   [ TESTING ]
                                          ↓
                                      Release
                          // bug bị chặn SỚM, chi phí sửa thấp hơn nhiều
