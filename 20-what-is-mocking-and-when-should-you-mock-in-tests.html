QUESTION 20 · TEST AUTOMATION

What is mocking, and when should you mock in tests?

Mocking là gì, và khi nào bạn nên mock trong test?

Sample Answer (English)

Mocking means replacing a real dependency, like a network call, a database, or a third-party service, with a fake implementation that returns controlled, predictable responses. In Playwright, I often use route interception with page.route() to intercept an API request and return a fixed mock response, without ever hitting the real backend. I mock whenever I want to isolate the unit under test, speed up execution, or avoid flaky third-party dependencies such as payment gateways or external APIs that are slow or unreliable. Mocking also lets me simulate edge cases, like server errors or empty states, that are hard to trigger for real. However, I avoid mocking in true end-to-end tests, because those are meant to verify the real integration between systems, and over-mocking there would hide real bugs and give false confidence.

🇻🇳 Bản dịch tiếng Việt

Mocking nghĩa là thay thế một dependency thật, chẳng hạn như một network call, một database, hoặc một dịch vụ bên thứ ba, bằng một cài đặt giả trả về các phản hồi có kiểm soát và có thể dự đoán được. Trong Playwright, tôi thường dùng route interception với page.route() để chặn một API request và trả về một mock response cố định, mà không bao giờ chạm vào backend thật. Tôi mock bất cứ khi nào tôi muốn cô lập unit đang được kiểm thử, tăng tốc độ chạy test, hoặc tránh các flaky third-party dependency như cổng thanh toán hoặc các API bên ngoài chậm và không ổn định. Mocking cũng cho phép tôi mô phỏng các edge case, như lỗi server hoặc trạng thái rỗng, những thứ khó tạo ra thật ngoài đời. Tuy nhiên, tôi tránh mocking trong các true end-to-end test, vì những test đó được thiết kế để xác minh sự tích hợp thật giữa các hệ thống, và việc over-mocking ở đó sẽ che giấu các lỗi thật và tạo ra sự tự tin giả tạo.

💻 Ví dụ minh họa (Playwright)

// Mock API response bằng page.route() — không gọi backend thật
await page.route('**/api/users/1', async route => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify({
      id: 1,
      name: 'Mock User',
      role: 'admin'
    }),
  });
});

// Điều hướng — UI sẽ hiển thị dữ liệu giả ở trên
await page.goto('/profile');
await expect(page.getByText('Mock User')).toBeVisible();

// Mô phỏng lỗi server để test edge case
await page.route('**/api/orders', route =>
  route.fulfill({ status: 500, body: 'Server Error' })
);

✅ Khi nào nên mock vs. dùng dependency thật

Tình huốngNên mock hay dùng thật?
Unit test một hàm/component riêng lẻMock — cô lập unit, không phụ thuộc hệ thống ngoài
Third-party API chậm, tốn phí, hoặc không ổn định (payment gateway...)Mock — tránh flaky và giảm chi phí gọi thật
Cần test edge case: lỗi 500, timeout, response rỗngMock — khó tạo tình huống này với hệ thống thật
Chạy test song song, cần nhanh và ổn định (CI)Mock — loại bỏ độ trễ mạng và dữ liệu thay đổi
True end-to-end test xác minh tích hợp toàn hệ thốngDùng thật — mock ở đây sẽ che giấu lỗi tích hợp thật
Smoke test trên môi trường staging/productionDùng thật — cần xác nhận hệ thống thật hoạt động đúng
