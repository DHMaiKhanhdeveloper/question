QUESTION 17 · TEST AUTOMATION

How do you run tests in parallel, and why does it matter?

Bạn chạy test song song (parallel) như thế nào, và tại sao nó quan trọng?

Sample Answer (English)

To run tests in parallel, I rely on Playwright's built-in worker processes and, in CI, its sharding feature. Locally, Playwright spins up multiple workers, each running test files in its own isolated browser context, so several suites execute at the same time instead of one after another. On CI, I split the full suite into shards across multiple machines using the shard flag, and each shard reports its results back so they can be merged into one report. Parallel execution matters mainly for faster feedback: a suite that takes forty minutes serially might finish in five minutes across eight workers, which keeps the CI pipeline fast and developers unblocked. The tricky part is that parallel tests must be fully independent, so I make sure each test creates its own test data, avoids shared state like global variables or shared files, and never depends on execution order or on another test's side effects. Without that isolation, parallel runs become flaky and unreliable.

🇻🇳 Bản dịch tiếng Việt

Để chạy test song song, tôi dựa vào cơ chế worker process có sẵn của Playwright và, trên CI, tính năng sharding của nó. Ở máy local, Playwright khởi tạo nhiều worker, mỗi worker chạy các file test trong browser context riêng biệt và cô lập, nhờ vậy nhiều bộ test chạy cùng lúc thay vì tuần tự. Trên CI, tôi chia toàn bộ bộ test thành các shard trên nhiều máy bằng cờ shard, và mỗi shard trả kết quả về để gộp lại thành một báo cáo. Việc chạy song song quan trọng chủ yếu vì phản hồi nhanh hơn: một bộ test chạy tuần tự mất bốn mươi phút có thể chỉ mất năm phút khi chạy trên tám worker, giúp CI pipeline nhanh và không làm chậm developer. Điểm khó là các test chạy song song phải hoàn toàn độc lập, nên tôi đảm bảo mỗi test tự tạo dữ liệu test riêng, tránh trạng thái dùng chung như biến toàn cục hay file dùng chung, và không phụ thuộc vào thứ tự chạy hay tác động phụ của test khác. Nếu thiếu sự cô lập đó, các lần chạy song song sẽ trở nên chập chờn (flaky) và không đáng tin cậy.

💻 Ví dụ minh họa (Playwright)

// playwright.config.js — cấu hình chạy song song + sharding
import { defineConfig } from '@playwright/test';

export default defineConfig({
  // Bật chạy song song trong cùng 1 file test
  fullyParallel: true,

  // Số worker chạy đồng thời (mặc định = số CPU core)
  workers: process.env.CI ? 4 : undefined,

  // Không cho phép .only lọt vào CI
  forbidOnly: !!process.env.CI,

  // Retry test flaky trên CI, không retry ở local
  retries: process.env.CI ? 2 : 0,
});

// Chạy sharding trên CI — chia bộ test cho nhiều máy
// Máy 1: npx playwright test --shard=1/4
// Máy 2: npx playwright test --shard=2/4
// Máy 3: npx playwright test --shard=3/4
// Máy 4: npx playwright test --shard=4/4

// Mỗi test tự tạo dữ liệu riêng — tránh shared state
test('creates its own user', async ({ page }) => {
  const uniqueEmail = `user_${Date.now()}@test.com`;
  await page.goto('/signup');
  // ... đăng ký với email độc nhất, không đụng dữ liệu test khác
});

✅ Điều kiện để chạy song song an toàn

Điều kiệnGiải thích
Isolated test data (dữ liệu cô lập)Mỗi test tạo dữ liệu riêng (email, id độc nhất) thay vì dùng chung 1 bản ghi
Independent browser contextsMỗi worker có context/session riêng — cookie, storage không ảnh hưởng lẫn nhau
Avoid global state (tránh trạng thái toàn cục)Không dùng biến module-level dùng chung giữa các test đang chạy song song
Unique file namesFile upload/screenshot/log phải đặt tên độc nhất để tránh ghi đè giữa các worker
No execution-order dependencyTest không được giả định chạy sau/trước một test cụ thể nào khác
