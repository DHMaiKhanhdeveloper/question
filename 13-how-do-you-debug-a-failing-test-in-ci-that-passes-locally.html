QUESTION 13 · TEST AUTOMATION

How do you debug a failing test in CI that passes locally?

Bạn debug một test bị fail trên CI nhưng lại pass ở local như thế nào?

Sample Answer (English)

First, I check the CI logs and any artifacts, like screenshots, videos, or traces, that the pipeline captured on failure. Those usually show exactly what the page looked like at the moment of failure. Next, I look for environment differences between CI and my local machine, such as different environment variables, base URLs, browser versions, or missing test data and seeded state. A common culprit is timing and flakiness, since CI runners are often slower and more resource constrained, so a race condition that never shows up locally can surface under load. I also consider headless versus headed rendering, since some elements behave differently without a real display, and network conditions, since CI may have higher latency. If I can't reproduce it, I try running the suite locally in headless mode with the same parallelism and throttled network to match CI as closely as possible.

🇻🇳 Bản dịch tiếng Việt

Đầu tiên, tôi kiểm tra log của CI (CI logs) và các artifact mà pipeline đã lưu lại khi fail, như screenshot, video hoặc trace. Những thứ đó thường cho thấy chính xác giao diện trông như thế nào tại thời điểm test fail. Tiếp theo, tôi tìm sự khác biệt môi trường (environment differences) giữa CI và máy local của mình, chẳng hạn như biến môi trường khác nhau, base URL, phiên bản trình duyệt, hoặc thiếu dữ liệu test và trạng thái đã được seed. Một nguyên nhân phổ biến là vấn đề timing và flakiness, vì các máy chạy CI (CI runners) thường chậm hơn và bị giới hạn tài nguyên hơn, nên một race condition không bao giờ xuất hiện ở local có thể lộ ra khi chịu tải. Tôi cũng xem xét chế độ headless so với headed rendering, vì một số phần tử hoạt động khác khi không có màn hình thật, và điều kiện mạng (network conditions), vì CI có thể có độ trễ cao hơn. Nếu không thể tái hiện lỗi, tôi thử chạy bộ test ở local với chế độ headless, cùng mức độ chạy song song (parallelism) và giới hạn băng thông mạng để giống với CI nhất có thể.

💻 Ví dụ minh họa (Playwright config để debug CI)

// playwright.config.js — bật artifact khi fail để debug CI
export default {
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 1 : undefined,
  use: {
    // chỉ lưu trace/video/screenshot khi test fail để đỡ tốn dung lượng
    trace: 'retain-on-failure',
    video: 'retain-on-failure',
    screenshot: 'only-on-failure',
    headless: true,
  },
  reporter: [['html', { open: 'never' }], ['list']],
};

// Trong CI job — upload artifact để tải về xem sau khi fail
// - name: Upload test artifacts
//   if: failure()
//   uses: actions/upload-artifact@v4
//   with:
//     name: playwright-report
//     path: playwright-report/

🔍 Nguyên nhân phổ biến: "Pass ở local, fail trên CI"

Nguyên nhânGiải thích
Timing / race conditionCI thường chậm hơn máy local, wait cố định hoặc thiếu chờ đúng điều kiện dễ gây fail
Environment config khác nhauBiến môi trường, base URL, feature flag, dữ liệu seed không giống local
Headless renderingFont, kích thước viewport, animation render khác khi chạy headless không có màn hình thật
Parallel test interferenceChạy song song trên CI khiến các test dùng chung dữ liệu/tài nguyên gây xung đột
Resource limitsCPU/RAM giới hạn trên CI runner làm chậm ứng dụng, dễ timeout hơn local
