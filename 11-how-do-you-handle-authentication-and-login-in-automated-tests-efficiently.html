QUESTION 11 · TEST AUTOMATION

How do you handle authentication and login in automated tests efficiently?

Bạn xử lý authentication/login trong automated test một cách hiệu quả như thế nào?

Sample Answer (English)

I try to avoid logging in through the UI for every single test, because it is slow and adds unnecessary flakiness. In Playwright, my preferred approach is to log in once in a global setup script, then save the authenticated session as a storageState.json file, including cookies and local storage. Every test that needs to be logged in simply reuses that storage state, so it starts already authenticated, skipping the login form entirely. When UI login truly needs to be tested, I keep that as a small, separate suite. For other cases, I sometimes call the login API directly to grab a token and inject it into the browser context. This keeps the test suite fast, reliable, and focused on the actual feature being tested instead of repeating the same login steps.

🇻🇳 Bản dịch tiếng Việt

Tôi cố gắng tránh việc đăng nhập qua giao diện người dùng (UI) cho mỗi test, vì nó chậm và gây ra sự không ổn định (flakiness) không cần thiết. Trong Playwright, cách tiếp cận tôi ưa thích là đăng nhập một lần trong một script thiết lập toàn cục, sau đó lưu phiên đã xác thực thành một file storageState.json, bao gồm cookie và local storage. Mỗi test cần đăng nhập chỉ đơn giản tái sử dụng storage state đó, nên nó bắt đầu ở trạng thái đã được xác thực sẵn, bỏ qua hoàn toàn form đăng nhập. Khi thực sự cần kiểm thử việc đăng nhập qua UI, tôi giữ nó như một bộ test riêng biệt. Với các trường hợp khác, đôi khi tôi gọi trực tiếp API đăng nhập để lấy token và tiêm nó vào browser context. Điều này giúp bộ test nhanh, đáng tin cậy, và tập trung vào tính năng thực sự đang được kiểm thử thay vì lặp lại các bước đăng nhập.

💻 Ví dụ minh họa (Playwright)

// global-setup.js — đăng nhập 1 lần, lưu session ra file
import { chromium } from '@playwright/test';

async function globalSetup() {
  const browser = await chromium.launch();
  const page = await browser.newPage();

  await page.goto('https://app.example.com/login');
  await page.getByTestId('username').fill('qa_user');
  await page.getByTestId('password').fill('S3cret!');
  await page.getByRole('button', { name: 'Login' }).click();
  await page.waitForURL('**/dashboard');

  // Lưu cookies + localStorage vào file dùng chung cho mọi test
  await page.context().storageState({ path: 'storageState.json' });
  await browser.close();
}

export default globalSetup;

// playwright.config.js
export default {
  globalSetup: require.resolve('./global-setup'),
  use: { storageState: 'storageState.json' }, // mọi test tự động "đã login"
};

// dashboard.spec.js — không cần login lại
test('shows welcome banner', async ({ page }) => {
  await page.goto('/dashboard'); // đã authenticated sẵn
  await expect(page.getByText('Welcome back')).toBeVisible();
});

⚖️ So sánh các cách xử lý login

Cách tiếp cậnƯu điểmNhược điểm
UI login mỗi test Kiểm thử luồng login thực tế end-to-end Rất chậm, tốn tài nguyên, dễ flaky khi UI/network chậm
API login + inject token Nhanh hơn UI, bỏ qua render form, vẫn linh hoạt cho nhiều role Cần biết/duy trì endpoint API, không kiểm thử UI login
storageState reuse (global setup) Nhanh nhất, login đúng 1 lần cho cả suite, ổn định cao Cần refresh khi session hết hạn; không phù hợp để test chính luồng login
